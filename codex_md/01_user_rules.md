# User Rules

## 文档规则

写 markdown 时，不要记录 worklog。

要记录的是：

- 我们改了什么。
- 为什么这么改。
- 当前系统的真实约束是什么。
- 下一次 session 接手时必须知道的上下文。

不要记录的是：

- 纯时间顺序的流水账。
- 没有长期价值的 debug 过程。
- 临时试错细节。

debug 过程如果已经没有长期价值，可以去掉。

## 协作规则

默认要先判断“当前框架是否已经存在”，不要把已经搭起来的东西误判成从零开始。

做判断时，优先关注：

- 主链路是否存在。
- 各脚本之间的输入输出是否能对上。
- 缺的是能力本身，还是工程一致性。

如果发现不匹配，要明确区分：

- 这是框架没搭起来。
- 还是框架已经有了，但接口和路径没统一。

## 改动规则

改动说明要尽量写成长期有效的项目知识，而不是一次性操作说明。

优先保留：

- 任务定义
- 数据格式约定
- 模块职责
- 当前已知问题
- 下一步应该收口的地方

少写或不写：

- 无关输出
- 临时命令回显
- 一次性机器状态

## 目标导向

这个项目当前更需要的是收口和对齐。

所以后续很多改动都应围绕这些问题展开：

- 参数统一
- 路径统一
- 训练 / merge / eval 接口统一
- 端到端可复现

不是优先追求“再多写几个脚本”。
